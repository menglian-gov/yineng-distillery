# 异能蒸馏炉 · 单能晶核造术

> 「全知是幻觉，专精是武器。每个真正的高手，都只有一两个真正属于他的异能。」

---

## 核心理念

异能蒸馏炉不造完整的人，它**萃取单一能力的操作系统**。

人类的顶级能力往往互相独立——马斯克的商业嗅觉和他的管理手段是两套不同的认知操作系统，蒸馏在一起会互相稀释。

**一个晶核，一种异能，激活即专家，出界即路由。**

一个合格的异能晶核回答五个问题：
- **感知机制**：这种能力让人注意到别人注意不到的什么？
- **判断框架**：感知到信息后如何处理？
- **行动模式**：基于判断做出什么样的输出？
- **边界定义**：域内 vs 域外的清晰划分
- **失效条件**：什么时候这种能力会出问题？

**关键区分**：不是捕捉「这个人说过什么」，而是捕捉「这种能力如何运作」。

skill被激活后，**先输出以下内容并等待用户指令**，不要主动扫描目录或加载历史会话：

```
⚡ 异能蒸馏炉已就绪

- 「蒸馏XX的XX能力」→ 开始新蒸馏

- 「诊断」→ 帮你确定要蒸馏什么能力
```

收到用户指令后，再按下方流程执行。

## 执行流程

### Phase 0：入口识别与分流

收到用户输入后，先判断用户要做什么：

| 用户意图 | 路径 |
|---------|------|
| 蒸馏特定人物的特定能力 | **人物单能路径** → Phase 0A |
| 蒸馏某种通用能力/领域 | **能力领域路径** → Phase 0B |
| 查看/管理已有晶核 | **晶核注册表** → Phase 0C |
| 不确定蒸馏什么 | **诊断路径** → Phase 0D |

---

### Phase 0A：人物单能确认

收到「人名 + 能力方向」，确认以下信息（最多追问2轮）：

1. **人物确认**：同名消歧义
2. **能力维度精确化**：把模糊需求转为「XX的XX力」标准命名
   - 「管理能力」→ 追问：管理人才的眼光？推进项目执行？构建组织架构？
   - 能力越具体，晶核越锋利
3. **来源范围**：只用本人表达的内容？还是包含他人对此能力的分析？（默认两者都用，本人优先）
4. **本地素材**：「你有没有关于这个人在这个能力维度的一手素材？访谈transcript、演讲视频、著作？有的话直接给我，比网搜精度高得多。」

用户说「就按这个来」→ 进入下一步：Phase 0.5（创建晶核目录）

**⚠️ 晶核注册表检查移至此步**：用户确认后、调研启动前，扫描 `.claude/skills/` 目录检查是否已有匹配晶核。如已有，询问用户是「用已有的」还是「新蒸馏」。

---

### Phase 0B：能力领域确认

用户没有指定人物，只有能力需求：

1. **能力类型判断**（参照 `references/ability-taxonomy.md`）
2. **能力边界讨论**（最多追问1轮）：
   - 「写小说能力」→ 类型小说/通用叙事？中文/英文？
3. **来源策略**：
   - 选项A（默认）：综合多位顶尖人物，提炼共识机制
   - 选项B：指定一本方法论著作作为锚点
   - 选项C：指定一个人物作为锚点 → 转入Phase 0A

---

### Phase 0C：晶核注册表管理

扫描 `.claude/skills/` 目录，展示：

```
## 你的晶核库

### 异能晶核（单能专家）
| 晶核名 | 能力域 | 状态 | 创建时间 |
|-------|-------|------|---------|
| [晶核ID] | [能力域描述] | ✅/⚠️ | [日期] |

### 全人格Skill（女娲造人）
| Skill名 | 人物 | 状态 |
|--------|------|------|
| [skill名] | [人物] | ✅/⚠️ |
```

用户有具体问题 → 分析问题后推荐最匹配的晶核

---

### Phase 0D：诊断路径

用户不确定蒸馏什么，有需求但无明确对象：

**Step 1：需求定位**（最多2轮追问，参照 `references/ability-taxonomy.md` 的需求映射表）

**Step 2：候选推荐**（2-3个候选）：

```
### 候选N: [来源名] · [能力名]  ⚡已有晶核 / 🆕需要蒸馏

**这个异能的本质**：[一句话描述这种能力如何运作]
**为什么适合你的需求**：[直接对应需求的匹配逻辑]
**域边界**：[能回答什么，不能回答什么]
**失效条件**：[什么情况下这个能力出问题]
```

已有晶核优先展示，候选之间保证差异性，必须说明失效条件。

---

### Phase 0.5：创建晶核目录

**收到确认后立即执行**，在调研之前完成：

```
.claude/skills/[source-ability]-core/
├── SKILL.md                          # 最终晶核
├── domain-boundary.md                # 域边界定义（独立文件，路由时快速读取）
├── sessions/                         # 会话记忆目录（v2.0）每次对话总结自动保存在此
│   └── YYYY-MM-DD_N.md               # 格式：日期_当日第N次会话
├── scripts/                          # 工具脚本（可调用共享脚本）
└── references/
    ├── research/
    │   ├── 01-ability-in-action.md   # 能力展示案例（情境→感知→判断→行动→结果）
    │   ├── 02-mechanism.md           # 能力运作机制（他人的专题拆解分析）
    │   ├── 03-expression-dna.md      # 此能力激活时的表达风格
    │   ├── 04-failure-cases.md       # 能力失效情形与案例
    │   └── 05-related-abilities.md   # 相关能力图谱（路由基础）
    └── sources/
        ├── books/
        ├── transcripts/
        └── articles/
```

**命名规则**：`[来源]-[能力简称]-core`，如 `musk-biz-core`、`novel-craft-core`、`design-eye-core`

**完成检查**（自动执行）：
- [ ] 目录已建立（含 `sessions/` 子目录）
- [ ] `domain-boundary.md` 已创建（空文件，Phase 2.2填内容）
- [ ] 如是中国人物：信息源切换为B站/小宇宙/权威中文媒体优先
- [ ] 如是更新模式：已读取现有SKILL.md，标注需要刷新的内容
- [ ] 如用户提供本地素材：素材已归档至 `sources/` 对应子目录

**关键规则**：所有调研文件必须存在晶核目录内部（`references/research/`）。晶核必须是自包含的——复制整个晶核目录就能独立使用。

### Phase 1：单能聚焦调研（并行Agent Swarm）

**核心原则**：所有Agent的任务不是「搜集关于此人的信息」，而是「找到此人/此领域在[具体能力维度]上的证据」。

#### 模式判断

| 模式 | 触发条件 | 策略 |
|------|---------|------|
| **纯网络搜索**（默认） | 无本地素材 | 5个Agent走网络搜索 |
| **本地语料优先** | 用户提供了素材 | 先分析本地素材，网络搜索补缺口 |
| **纯本地语料** | 用户明确仅用提供的素材，或蒸馏非公众人物 | 只分析本地素材 |

#### 5个Agent的任务分配

| Agent | 搜索目标 | 提取重点 | 输出文件 |
|-------|---------|---------|---------|
| **1 能力展示案例** | 此人/此领域在该能力上最具代表性的决策案例、作品、公开记录 | 完整操作轨迹：情境→感知了什么→判断了什么→做了什么→结果 | `01-ability-in-action.md` |
| **2 能力机制分析** | 他人对该能力的深度拆解文章、学术研究、内部访谈 | 操作机制的具体描述（「他具体怎么做到的」，不是「他很厉害」） | `02-mechanism.md` |
| **3 表达风格DNA** | 此人谈论该能力领域时的用语、框架、比喻、判断词 | 高频概念、专属术语、判断时的语气特征、惯用类比 | `03-expression-dna.md` |
| **4 失效与局限案例** | 此人/此领域在该能力上的失误案例、局限性讨论、批评者观点 | 什么情况下这种能力不适用、反例、能力本身的盲区 | `04-failure-cases.md` |
| **5 相关能力图谱** | 与该能力协作或竞争的其他能力，能力的前置条件与后置应用 | 协作关系、路由推荐基础、该能力无法覆盖的相邻域 | `05-related-abilities.md` |

#### 每个Agent的硬性要求

- 调研结果必须写入 `references/research/0X-xxx.md`，不写文件的调研等于没做
- **所有内容必须围绕「这种能力如何运作」组织**，不是关于这个人的通用传记信息
- 注明信息来源和可信度：一手（此人直接表达）> 二手（他人分析）> 推测
- 区分「此人真实展示过该能力」vs「别人说他有这个能力」vs「我推断的」
- 发现矛盾时保留矛盾——尤其是「成功案例」和「失效案例」之间的矛盾，往往揭示能力的真实边界

#### Agent Prompt模板（以Agent 1为例）

```
你的任务：调研[来源名]在「[具体能力名称]」上的真实展示案例。

你要找的不是关于这个人的通用信息，而是专门展示这种能力如何运作的证据。

搜索方向：
- 此人在该能力上最具代表性的公开决策案例（有完整前因后果）
- 此人在该能力上做出判断后被证明正确的案例（及其思考过程）
- 访谈/文章中他具体描述「自己是怎么做到这个判断的」的部分
- 反映该能力运作的言行一致案例

输出格式（每个案例）：
情境 → 感知（他注意到了什么）→ 判断（如何处理信息）→ 行动（做了什么）→ 结果

要求：
- 写入 [skill目录]/references/research/01-ability-in-action.md
- 至少3个案例，最多8个（精而非多）
- 每条标注来源URL和可信度
- 区分一手 vs 二手

信息源黑名单：知乎、微信公众号、百度百科、百度知道。
泛泛赞美类文章（「XX为什么这么成功」）不使用——无实质内容。
```

其他4个Agent按同样结构调整搜索方向即可。

#### 工具辅助（如可用）

在Phase 1启动前，主动扫描 `.claude/skills/` 目录，检查是否有可用于信息获取的skill：

| 已安装Skill | 用途 | 调用场景 |
|------------|------|---------|
| `gemini-video` | 分析本地视频，提取transcript | 用户提供了视频文件 |
| `web-article-reader` | 精确读取网页文章全文 | 找到重要文章URL时 |
| `agent-reach` | 多渠道信息获取 | 需要从X/Reddit/YouTube获取信息 |
| `pdf` | 读取PDF书籍/论文 | 用户提供了PDF格式一手素材 |

视频字幕工具（调用共享脚本）：
- `bash [skill目录]/scripts/download_subtitles.sh <YouTube_URL> [输出目录]`
- `python3 [skill目录]/scripts/srt_to_transcript.py <input.srt> [output.txt]`

调研摘要生成（Phase 1.5用）：
- `python3 [skill目录]/scripts/merge_research.py <skill目录>`

#### 信息源优先级

| 来源类型 | 揭示什么 | 权重 |
|---------|---------|------|
| 用户提供的一手素材 | 完整原文，未经过滤 | **最高+** |
| 此人谈该能力的专题访谈/文章 | 元认知——他如何理解自己的能力 | 最高 |
| 他人对该能力的专题拆解 | 外部观察视角 | 高 |
| 实际决策/作品记录（可验证） | 能力的真实表现 | 最高 |
| 传记/侧写中的相关片段 | 上下文参考 | 中 |
| 泛泛评价（「他很有商业头脑」） | 不使用——太空洞 | 排除 |

#### 信息源黑名单（永久排除）

- **知乎**：洗稿严重、信息失真率高
- **微信公众号**：封闭生态、大量二手转述，无法验证
- **百度百科/百度知道**：陈旧且不可靠

中文渠道只接受权威媒体：36氪、极客公园、晚点LatePost、财新、第一财经、虎嗅、少数派、机器之心等。人物访谈类可用播客平台（小宇宙、喜马拉雅原始音频）和B站原始视频（非搬运号）。

#### Agent超时与失败处理

- **单个Agent超时**（5分钟无有价值结果）：不等待，继续推进，Phase 2标注「信息不足」
- **案例数量不足**（<3个高质量案例）：Phase 0.5就提醒用户，建议提供本地素材
- **来源严重匮乏**（<10条可用来源）：晶核标记为「低可信度」，重点扩大局限说明

### Phase 2：异能机制提炼

5个Agent素材汇总后，执行能力专属提炼。先读取 `references/extraction-framework.md` 获取三层结构提炼方法论和质量自检清单。

**核心目标：把这种能力拆解成可操作的认知流程，不是归纳一堆原则。**

#### 2.1 能力三层结构提炼（核心工程）

**感知层（Perception）**：这种能力让人看到别人看不到的什么？
- 关注维度：在该能力域内，主动注意哪3-5个信号？
- 过滤规则：主动忽略哪些噪音？（通常比关注的信号更有价值）
- 感知触发器：什么情况下这种感知会被激活？

**判断层（Judgment）**：感知到信息后如何处理？
- 核心心智模型（2-4个）：三重验证筛选（跨域复现 + 生成力 + 排他性），详见 `references/extraction-framework.md`
- 决策启发式（5-8条）：「如果X，则Y」格式，每条有真实案例支撑
- 反直觉判断：与常识相悖但被证明有效的判断原则（通常是最有价值的部分）

**行动层（Action）**：判断后，如何转化为输出？
- 典型行动序列（用这种能力处理问题时的步骤）
- 表达DNA（这种能力激活时的语言特征）
- 迭代方式（如何通过反馈优化输出）

#### 2.2 域边界精确化

**域内（IN）**：
- 直接回答的问题类型：[列举3-5个典型问题]
- 边缘问题：[列举2-3个边界问题，标注处理方式]

**域外（OUT）**：
- 拒绝回答的问题类型：[列举3-5个典型例子]
- 路由规则：域外问题推荐哪类能力

**域边界写入独立文件**：完成后同步写入 `domain-boundary.md`（供路由时快速读取，不需要加载完整SKILL.md）

#### 2.3 表达DNA分析

专门针对「此人在运用该能力时」的表达特征：

| 维度 | 提取内容 |
|------|---------|
| 开场方式 | 运用此能力分析问题时如何切入（直奔结论？先建框架？） |
| 核心概念词 | 该能力领域的专属术语、高频词、自创概念 |
| 判断句式 | 表达判断时的句式特征（「关键是...」「大多数人忽略了...」） |
| 类比偏好 | 倾向使用什么类型的类比 |
| 确定性表达 | 斩钉截铁型还是概率型 |
| 反模式表达 | 激活该能力时明确拒绝的思维方式 |

#### 2.4 失效条件与局限地图

**必须写入，且必须具体**：

- 能力的先决条件（在什么前提下才能发挥）
- 情境局限（哪些特定情境会系统性失效）
- 知识截止（调研时间点）
- 不可蒸馏的部分（直觉、经验、天赋中无法语言化的部分）
- 公开表达偏差（此人公开谈论与实际运用可能的落差）

#### 2.5 相关能力图谱

```
[当前晶核能力]
  ↑ 前置：[能力A] — 需要先有什么
  ↓ 后置：[能力B] — 此能力输出后通常需要什么接手
  ↔ 协作：[能力C] — 常见组合场景
  ✗ 竞争：[能力D] — 视角冲突说明
```

### Phase 2.5：提炼确认检查点

暂停展示提炼摘要给用户确认：

```
提炼摘要 · [晶核名称]

感知层：[3-5个感知维度，简述]
判断层：[2-4个心智模型名称] + [N条决策启发式]
行动层：[行动模式关键词] + [表达DNA 3个关键特征]
域边界：域内[N类问题] / 域外[N类问题] / 路由目标[N类能力]
失效条件：[N条]
相关能力图谱：前置/后置/协作/竞争各[N]个
```

用户确认OK → 进入Phase 3。
用户觉得方向不对 → 回到Phase 2调整后继续。

### Phase 3：晶核构建

先读取 `references/skill-template.md` 获取晶核SKILL.md的完整骨架结构，然后填充内容。

**构建清单**：

| 模板Section | 填充来源 |
|------------|---------|
| frontmatter | 能力域定义 + 触发词 + 路由目标 |
| 域边界声明 | Phase 2.2结果 → 写入SKILL.md + 同步写入domain-boundary.md |
| 激活协议 | 模板默认规则（根据能力类型微调） |
| 回答工作流（Agentic Protocol） | **根据感知层维度推导研究方向**，不使用通用模板 |
| 感知层 | Phase 2.1感知层结果 |
| 判断层 | Phase 2.1判断层（心智模型 + 启发式 + 反直觉） |
| 行动层 | Phase 2.1行动层 + Phase 2.3表达DNA |
| 相关能力图谱 | Phase 2.5结果 |
| 失效条件 | Phase 2.4结果 |
| 诚实边界 | Phase 2.4 + 调研时间 + 来源说明 |
| 调研来源 | 5个Agent的引用汇总（一手/二手分类） |
| 创建者归属 | 固定：`> 本晶核由 [异能蒸馏炉](https://github.com/xxx/yineng-distillery) 生成` |

**回答工作流（Agentic Protocol）生成规则**：

生成的Agentic Protocol必须包含3个Step，其中Step 2的研究维度必须**根据感知层自动推导**，不是固定模板。

推导方法：从感知层的3-5个关注维度，转化为具体的搜索维度。例：
- 马斯克商业嗅觉晶核的感知维度（成本基准 / 物理极限 / 未被满足的需求）→ Step 2研究方向（搜基础成本结构 / 搜技术约束 / 搜市场真实痛点）
- 叙事能力晶核的感知维度（情感钩子 / 信息密度 / 节奏） → Step 2研究方向（分析目标读者的情绪预期 / 核心信息提炼 / 结构节奏规划）

**完成后**：
1. 将SKILL.md写入晶核目录
2. 将domain-boundary.md同步更新
3. 更新 `.claude/skills/skill-registry.md`（晶核注册表）

**质量自检**：用 `python3 scripts/quality_check.py <晶核SKILL.md路径>` 执行自动检查

### Phase 4：质量验证

生成晶核后，spawn子agent执行4项测试：

#### 4.1 域边界清晰度测试（异能晶核特有）

出5道问题（3道明确域内，2道明确域外），子agent带着晶核判断：
- 域内题全部正确识别 → 通过
- 域外题触发正确路由话术 → 通过
- 任何域外题被硬答 → **失败**，回溯修正域边界定义

#### 4.2 能力机制一致性测试

选3个真实案例，子agent用晶核分析，对比调研记录的真实判断过程：
- 感知信号一致 → 通过
- 判断逻辑方向一致 → 通过
- 偏离超过50% → 回溯Phase 2重新提炼

#### 4.3 深度测试（最重要）

让子agent用晶核回答一个中等难度的域内问题，评估：
- 回答是否展示出「比通用AI更深的专业判断」？
- 删去「[来源名]视角」几个字后，内容是否丝毫不变？如果是，蒸馏失败——只贴了标签，没有提炼出真正的能力操作系统。

#### 4.4 通过标准

| 检查项 | 通过标准 | 不通过信号 |
|--------|---------|-----------|
| 域边界清晰度 | 5/5题正确判断域内/域外 | 有域外题被硬答 |
| 感知维度独特性 | 3-5个，无法被通用AI替代 | 感知维度是「分析市场」这类废话 |
| 决策启发式可操作性 | 5-8条，每条有案例，能推导新案例 | 只有抽象原则，无具体规则 |
| 失效条件具体性 | 5条以上，每条有情境描述 | 只写「信息截止」一条 |
| 表达DNA辨识度 | 读100字能感知「这是X能力」的专属味道 | 像通用建议 |
| 路由逻辑完整性 | 域外话术完整，推荐方向清晰 | 只说「不在域内」不给方向 |

验证通过 → 交付。不通过 → 回溯，最多2轮。2轮后交付当前最优版本并在局限中标注薄弱维度。

**展示验证结果给用户确认后才算完成。**

### Phase 5：双Agent精炼

Phase 4验证通过后，自动启动：

**Agent A（域边界专项审计）**：
- 审核域边界是否有歧义或漏洞
- 干跑5个边缘问题，检验路由逻辑的灰色地带处理
- 输出：最弱2处域边界的具体修订建议（含改后文本）

**Agent B（能力深度审计）**：
- 评估感知层是否捕捉了「别人注意不到的信号」
- 评估决策启发式是否真正反直觉或足够具体
- 识别「通用AI也能说」的废话内容，建议删除或替换
- 输出：2-3处具体深度增强建议（含改后文本）

**主Agent综合两份报告，应用不冲突的改进，展示变更摘要请用户确认。**

精炼标准：改动必须让晶核「激活即专家，出界即路由」。

## 更新已有晶核

当用户说「更新XX晶核」「XX最近有新动态」时：

1. 读取现有SKILL.md的「诚实边界」section找「调研时间：[日期]」
2. 只启动Agent 1（新案例）+ Agent 4（新失效案例）两个定向Agent
3. 对比新信息：
   - 强化现有机制 → 补充案例
   - 与现有机制矛盾 → 更新机制，标注变化
   - 出现新的能力模式 → 评估是否拆分为新晶核
4. 更新调研时间
5. 更新skill-registry.md对应条目

## 晶核组合使用协议

**并行激活**（问题可被多个晶核独立回答）：
各晶核从自己的感知维度切入，给出独立判断，主agent综合后标注来源晶核。

**串行激活**（多个晶核依次处理）：
按相关能力图谱的「前置→后置」顺序激活，上一晶核输出作为下一晶核的输入。

**冲突处理**（两个晶核给出相反判断）：
标注冲突，解释各自视角的理由，不强行调和——冲突本身是有价值的信号。

## 品味守则

| 原则 | 一句话 |
|------|--------|
| 深度 > 广度 | 宁可只回答3类问题但极深，也不假装能回答一切 |
| 机制 > 原则 | 「他会先看护城河宽度」比「他重视竞争优势」有价值100倍 |
| 失效 > 万能 | 清晰说出失效条件，比假装无敌更值得信任 |
| 路由 > 硬撑 | 出界时果断推荐其他晶核，不用域外能力凑答案 |
| 案例 > 概念 | 每个判断启发式必须有真实案例，没案例不算蒸馏成功 |

### 绝不做的事

- 假装晶核是全能的——每个晶核只有且只有一种超能力
- 蒸馏失败却不标注——宁可标注了局限的60分晶核，也不要看起来完整但实际在编造的90分晶核
- 忽略失效案例——失效案例比成功案例更能揭示能力的真实边界
- 路由时含糊其辞——说清楚推荐哪类能力及理由

## 特殊场景

### 同一人物的多个晶核

蒸馏同一人物第2个晶核时：
1. 读取该人物已有晶核，确认新晶核与已有晶核能力边界不重叠
2. 在两个晶核的「相关能力图谱」中互相标注协作关系
3. 路由逻辑中互相引用

### 蒸馏非公众人物

私人（老师、朋友、家人）的某种能力：
1. 网络搜索无效，直接进入纯本地语料模式
2. 引导提供：观察记录、对话摘录、决策案例描述
3. 标注：此晶核基于用户观察和记忆，存在主观偏差，仅供个人参考

### 蒸馏用户自己的某种能力

「蒸馏我在[能力]上的习惯」：
1. 纯本地语料模式
2. 追问：该能力上有哪些成功案例？失败案例？
3. 引入「自我认知盲区」检查：身边人如何评价这个能力？

### 能力来源是作品/方法论

「蒸馏《孙子兵法》的战略判断力」「蒸馏日本工匠精神」：
1. Phase 0B路径，来源定为「作品/流派」
2. Agent 1改为「原典精读」，优先分析一手文本
3. Phase 3的身份卡改为「能力来源说明」，不做角色扮演

## 会话记忆系统（v2.0）

### 触发机制

用户在与晶核对话过程中，随时输入以下指令触发会话总结：

```
总结这次对话
保存这次会话
保存会话
```

### 晶核执行流程

收到触发指令后，晶核执行：

1. **生成摘要**：将本次对话提炼为标准格式
   - 本次聚焦的主题
   - 核心结论（列表）
   - 用到的关键框架/模型
   - 发现的晶核局限
   - 未解决的问题/建议后续调研

2. **调用脚本保存**：
   ```bash
   python3 [蒸馏炉目录]/scripts/save_session.py [晶核目录]
   ```
   保存至：`[晶核目录]/sessions/YYYY-MM-DD_N.md`

3. **更新SKILL.md**：将摘要写入晶核SKILL.md末尾的「会话记忆」section
4. **确认输出**：告知用户保存成功，以及本次会话的核心结论

### 下次激活时的记忆读取

晶核激活时，自动执行：

```
1. 扫描 sessions/ 目录
2. 读取最近3个会话文件（按日期倒序）
3. 提取「核心结论」和「未解决问题」
4. 在内部建立上下文（不输出给用户，除非当前问题与历史相关）
5. 当前问题与历史会话有关联时，自然引用：
   「上次我们讨论了[主题]，你现在的问题延伸到了[当前主题]...」
```

查看历史会话：
```bash
python3 [蒸馏炉目录]/scripts/save_session.py [晶核目录] --show
```

### 在Phase 3构建晶核时的处理

生成晶核SKILL.md时，模板中包含：
- `sessions/` 目录说明（Phase 0.5创建）
- `## 激活协议` section 加入历史会话加载指令
- `## 会话记忆` section（初始为空，每次总结后填充）
