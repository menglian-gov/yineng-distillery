# 能力维度分类体系

> 蒸馏前确认能力类型，不同类型的蒸馏策略不同

## 一、能力类型全景图

| 能力类型 | 定义 | 典型案例 | 蒸馏重点 | 难度 |
|---------|------|---------|---------|------|

| **执行推进类** | 如何推动事情落地、管理过程、克服阻力 | 马斯克工程执行力、贝佐斯逆向工作法、稻盛和夫现场主义 | 操作流程 + 决策启发式 | ⭐⭐⭐ |
| **文字表达类** | 如何输出特定类型的文字内容 | 写小说能力、写公文能力、写销售文案、写深度报道 | 结构法则 + 风格DNA | ⭐⭐⭐ |
| **人际洞察类** | 如何理解人、判断人、影响人 | 看人懂人能力、谈判能力、销售能力、招聘判断力 | 观察维度 + 心理模型 | ⭐⭐⭐⭐⭐ |
| **设计审美类** | 如何感知和创造视觉/体验/空间品质 | 设计能力、产品体验直觉、品味判断力 | 审美标准 + 取舍原则 | ⭐⭐⭐⭐ |
| **教学传递类** | 如何把复杂知识变得可理解、可传递 | 费曼教学法、故事化解释能力、结构化表达 | 简化机制 + 类比策略 | ⭐⭐⭐ |
| **创意生成类** | 如何在特定约束下产出新颖有效的方案 | 广告创意能力、产品概念能力、叙事创意 | 生成触发器 + 筛选标准 | ⭐⭐⭐⭐ |

## 二、需求到能力类型的映射

用户描述模糊需求时，用此表反推蒸馏方向：

| 用户表达 | 可能的能力类型 | 追问方向 |
|---------|--------------|---------|
| 「想做更好的商业决策」 | 认知判断类 | 商业判断 vs 投资判断？宏观市场 vs 具体业务？ |
| 「想识别好的人/人才」 | 人际洞察类 | 识别人才？理解用户？谈判？ |
| 「想写出更好的文章」 | 文字表达类 | 什么类型的文章？目标读者？商业文章 vs 创意写作？ |
| 「想做出更好的产品/设计」 | 设计审美类 | 用户体验设计？视觉设计？产品功能决策？ |
| 「想把复杂的事说清楚」 | 教学传递类 | 对谁说清楚？什么场景？写作 vs 口头讲解？ |
| 「想在不确定中做决策」 | 风险判断类 | 投资决策？业务决策？人生决策？ |
| 「想找到正确的方向/路径」 | 策略规划类 | 职业路径？产品策略？竞争策略？ |
| 「想让内容更吸引人」 | 创意生成类 | 什么内容形式？什么平台？吸引谁？ |
| 「想推动事情真正落地」 | 执行推进类 | 自己执行？管理他人执行？克服什么阻力？ |
| 「想理解人为什么这样做」 | 人际洞察类 | 理解特定的人？理解用户群体？心理模型？ |

## 三、能力来源类型

蒸馏来源不一定是人，可以是：

### 3.1 具体人物

最常见的来源。信息最充足，但「公开表达 vs 真实运用」的偏差风险也最高。

**适合蒸馏的人物特征**：
- 留下大量可分析的公开素材（著作、访谈、决策记录）
- 在该能力维度上有多个可验证的真实案例
- 他人（批评者和支持者）对该能力都有分析和评论

**蒸馏难度**：
- 容易：有大量一手素材的西方公众人物（Musk、Feynman、Naval等）
- 中等：中国公众人物（信息渠道更分散，需要B站/小宇宙等专属渠道）
- 困难：非公众人物、低调者、刻意不公开该能力的人

### 3.2 方法论著作

如《孙子兵法》《穷查理宝典》《写作这回事》《结构化思维》等。

**优势**：一手信息质量最高，作者亲自阐述了能力框架
**局限**：书里的「建议」和作者真实的「实践」可能有落差；无真实案例

Agent分工变体：
- Agent 1改为「原典精读 + 案例提取」
- Agent 2改为「他人对此书方法论的实践验证」
- Agent 3改为「作者在其他场合阐述同一能力的语言特征」

### 3.3 领域流派/集体智慧

如「顶级风险投资人的项目判断力」「日本工匠精神中的品质感知力」「律师的论证能力」。

**优势**：能萃取领域共识，不依赖单一人物
**局限**：表达DNA无法模拟特定人物，只能用中性专业风格；可能稀释独特性

Agent分工变体：
- 先识别该领域3-5个核心代表人物
- 每人分配1-2个Agent（不需要6个Agent），聚焦在该能力维度上
- Phase 2提炼时：重点找「多人共同体现的机制」（≥3人都用 = 领域共识）

### 3.4 用户自己的某种能力

「我想把自己在[能力]上的思维方式沉淀下来」。

**适合场景**：用于个人知识管理、培训传承、团队标准化
**注意事项**：
- 纯本地素材模式，引导提供：决策记录、复盘笔记、案例描述
- 特别关注「自我认知盲区」——用户对自己能力的认知可能不准确
- 建议引入「他人观察」作为校验：「身边人怎么描述你在这方面的特点？」

## 四、能力粒度判断

**太宽（无效）** → 「思维能力」「领导力」「智慧」
- 症状：域边界无法清晰定义，几乎所有问题都在域内
- 解决：继续拆分，直到能说出「这种能力让人注意到X、忽略Y」

**合适（有效）** → 「马斯克的第一性原理成本判断力」「叙事结构与情绪节奏把控力」「招聘时的人才信号识别力」
- 症状：有清晰的感知维度，有明确的域内外边界，有可识别的失效条件

**太窄（价值低）** → 「马斯克分析特斯拉电池成本的具体方法」
- 症状：适用场景极少，大多数真实问题都在域外
- 解决：上升一个抽象层级，扩大适用范围但保持核心机制不变

**粒度判断口诀**：
> 能说清楚「注意什么、忽略什么、在什么情境下有效、在什么情境下失效」，粒度就合适。

## 五、同一人物多晶核的拆分原则

当用户想蒸馏同一人物的多种能力时，如何判断是否需要拆成多个晶核：

**需要拆分**（建独立晶核）：
- 感知维度完全不同（商业嗅觉感知「成本/市场空白」，管理能力感知「人的动机/阻力」）
- 应用场景基本不重叠
- 失效条件不同

**不需要拆分**（合并入一个晶核）：
- 两种「能力」其实是同一感知机制的两面
- 域边界几乎完全重叠
- 一起蒸馏后内容不会稀释

**典型拆分示例（马斯克）**：
| 晶核 | 感知重点 | 域内问题 |
|------|---------|---------|
| `musk-biz-core` 商业嗅觉 | 成本结构、物理极限、市场空白 | 「这个机会值得做吗？」「成本能降到多少？」 |
| `musk-exec-core` 执行推进 | 里程碑依赖、关键路径、人的motivation | 「如何推进这个项目？」「谁应该负责这个？」 |
| `musk-hiring-core` 人才判断 | 简历真实度、能力验证点、文化适配 | 「这个候选人怎么样？」「如何判断人的潜力？」 |

三个晶核的感知维度、域边界、失效条件完全不同，必须独立蒸馏。
